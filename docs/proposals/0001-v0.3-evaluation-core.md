# Proposal: v0.3 Evaluation Core

Status: Draft
Tracking issue: #6
Target release: v0.3.0
Created: 2026-04-24

## Summary

`go-eval` already has the core primitives for single-turn LLM evaluation in Go:
`Case`, `Judge`, `Metric`, `Runner`, LLM-as-judge metrics, deterministic
metrics, `Compound`, `Precheck`, JSONL result sinks, benchmarks, and an OpenAI
adapter.

This proposal defines the next small, coherent step: make `go-eval` feel like a
complete local-first evaluation core for Go without turning it into a full
evaluation platform.

The direction is to strengthen local, test-native evaluation around:

- conversation evaluation
- dataset and golden-case loading
- JSONL result comparison
- a small `goeval` command-line wrapper
- additional judge adapters
- a better getting-started path

## Motivation

Python projects have mature LLM evaluation tools, but Go projects often rely on
ad hoc scripts, external Python evaluation suites, or overly shallow
deterministic checks.

`go-eval` takes a different path: evaluations should live next to Go code and
run through the normal Go toolchain.

The current library already handles single-turn RAG-style evaluations well. The
remaining gap is workflow shape: teams need a simple way to define cases outside
test code, compare runs, evaluate conversations, and onboard quickly.

## Goals

- Add a first-class conversation evaluation model.
- Add a dataset/golden loader while keeping the core package stdlib-only.
- Add local regression comparison for JSONL result files.
- Add a minimal CLI that wraps the existing `go test` workflow.
- Add at least one non-OpenAI adapter.
- Improve documentation with a complete getting-started guide.
- Keep the project small, idiomatic, and local-first.

## Non-goals

- No hosted platform.
- No production tracing or observability in this release.
- No synthetic data generation.
- No red-teaming framework.
- No attempt to clone every metric from larger evaluation frameworks.
- No dependency-heavy core package.

## Proposed Design

### Conversation Evaluation

Add a conversation-oriented API for multi-turn evaluation.

Possible shape:

```go
type Message struct {
	Role    string
	Content string
}

type ConversationCase struct {
	Messages []Message
	Expected string
	Context  []string
	Metadata map[string]any
}

type ConversationMetric interface {
	Name() string
	ScoreConversation(context.Context, Judge, ConversationCase) (Result, error)
}
```

The exact names and package placement are open. The key requirement is that
conversation evaluation should still work naturally from `go test`.

### Dataset and Golden Loader

Add a loader for external test cases.

Initial JSON format:

```json
{
  "cases": [
    {
      "name": "france-capital",
      "input": "What is the capital of France?",
      "output": "Paris is the capital of France.",
      "expected": "Paris",
      "context": ["Paris is the capital of France."],
      "metadata": {
        "flow": "rag.answer",
        "tier": "critical",
        "dataset": "smoke/v1"
      }
    }
  ]
}
```

The JSON loader can live in the root package while remaining stdlib-only. YAML
support is deferred to a later subpackage or separate module if it adds a
dependency.

### JSONL Regression Comparison

Add a package or command that compares two result files generated by
`GOEVAL_RESULTS_DIR`.

Expected comparisons:

- score deltas by test and metric
- pass/fail regressions
- token deltas
- latency deltas
- changed compound dimensions

Possible command:

```bash
goeval compare old/results.jsonl new/results.jsonl
```

### Minimal CLI

Add a small command that preserves `go test` as the execution engine.

Possible commands:

```bash
goeval test ./...
goeval compare old.jsonl new.jsonl
goeval version
```

`goeval test` should primarily set `GOEVAL=1` and delegate to `go test`.

### Additional Adapters

Add one or more adapters while keeping the root package dependency-free.

Candidates:

- Ollama
- Anthropic
- Gemini

Each adapter should live in its own module under `adapters/`.

### Documentation

Add a complete getting-started guide that demonstrates:

- a deterministic local judge
- an OpenAI-backed judge
- RAG metrics
- deterministic metrics
- `Compound`
- `Precheck`
- JSONL result output
- benchmark usage

## Compatibility

The root package should remain compatible with the current v0.2 public API.

New functionality should be additive. Any dependency-heavy features should live
in subpackages or separate modules, preserving the stdlib-only core.

## Risks

- The scope could grow into a platform instead of a library.
- Dataset loading could pull dependencies into the core package if not isolated.
- A CLI could obscure the value of native `go test` integration if it becomes
  the primary execution engine.
- Conversation APIs may need revision after real use.

## Alternatives Considered

### One Large Evaluation Platform Clone

Rejected for now. Larger evaluation platforms include testing, datasets,
tracing, synthetic data, red-teaming, observability, and cloud workflows.
`go-eval` should first become excellent at local Go-native evaluation.

### Only Add More Metrics

Adding metrics is useful, but workflow gaps are currently more important:
datasets, conversations, comparisons, and docs unlock broader usage.

### No CLI

Using only `GOEVAL=1 go test ./...` is enough technically. A small CLI may still
help discoverability and make comparison workflows easier, as long as it remains
a wrapper instead of a new runtime.

## Open Questions

- Should conversation APIs live in the root package or a subpackage?
- Should dataset loading support YAML in v0.3, or begin with JSON only?
- Should `goeval compare` live in a `cmd/goeval` module or a separate package
  first?
- Which adapter should come after OpenAI?
- Should result comparison define a stable JSON output for CI?
- Should `Compound` be extended for weighted dimensions?

## Implementation Plan

1. Create tracking issues for each accepted workstream.
2. Design and implement conversation cases and metrics.
3. Add dataset loading with examples.
4. Add JSONL comparison package.
5. Add minimal `cmd/goeval`.
6. Add one additional adapter.
7. Write a getting-started guide.
8. Update `CHANGELOG.md` and release v0.3.0.
